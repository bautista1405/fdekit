# Demo Script

This demo should make FDEKit feel like a field-deployment workbench, not another agent wrapper.

## One-Line Story

An FDE joins a customer account, connects an AI support triage agent to customer systems, proves governance and evals, generates a stakeholder dashboard, then captures the successful workflow as a reusable recipe for the next customer.

## Run It

From the repo root:

```bash
npm install
npm run demo
```

For a project created from the published package, install the support-triage recipe first, then use the recipe's one-command runner:

```bash
npx fdekit recipe install support-triage
npm install
npm run demo
```

For repeated recordings after a successful build:

```bash
npm run demo:fast
```

The script:

1. Builds the monorepo.
2. Checks that scaffold-owned examples still match the recipe templates.
3. Starts the support-triage customer API if it is not already running.
4. Runs doctor and validation checks.
5. Runs the support triage agent loop.
6. Exports feedback into eval candidates.
7. Runs lower-level evals and macro evals.
8. Generates a report and local console.
9. Captures the working deployment as `support-renewal-risk` on demand.

After the script finishes, open the generated outputs:

```txt
examples/support-triage/.fdekit/console.html
examples/support-triage/recipes/support-renewal-risk/
```

Those paths are generated by `scripts/demo.mjs`; the checked-in example stays as a runnable source fixture.

### 0:20 - The Deployment Config

Open `examples/support-triage/fde.config.ts`.

Point at:

- `defineWorkflow()` for the customer workflow.
- `defineOutcomeMetric()` for measurable impact.
- `defineDataLayers()` for system of record, rules, raw intake, and feedback.
- `defineRollout()` for staged adoption.
- connectors for customer API, GitHub-style issues, and Slack-style messages.
- governance for scopes, audit, PII/secrets, budgets, and environment rules.
- eval assertions that make quality unavoidable.

Say:

"The config is not just model/provider selection. It is the deployment contract."

## What Not To Demo

- Do not lead with model selection.
- Do not spend time explaining every connector package.
- Do not show a generic chat UI.
- Do not over-explain monorepo internals.

The strongest demo is workflow -> governed run -> evidence -> dashboard -> captured recipe.

## Next Step

If you just ran the demo, use [Reference Architectures](./reference-architectures.md) to decide the next posture. Stay with the local deterministic demo for onboarding, move to hybrid when one live connector matters, or use the production-shaped setup when governance and handoff evidence are the real goal.
